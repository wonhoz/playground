### 1. Run.Deck — 프로젝트 원클릭 런처

매일 아침 반복하는 의식이 있다:
VS Code 열고, 터미널에서 `docker compose up`, 브라우저에서 `localhost:3000` 열고,
또 다른 터미널에서 `dotnet watch run`... 이걸 프로젝트마다, 매일, 수동으로 한다.

**Run.Deck은 이 "의식"을 레시피로 저장한다.**

```yaml
# 예시 레시피: "Playground 개발"
- open: "C:\Users\admin\source\repos\+Playground" → VS Code
- run:  "dotnet build Playground.sln"
- open: "https://localhost:5001" → 기본 브라우저
- run:  "docker compose up -d" @ "C:\Projects\backend"
```

한 번 클릭하면 순서대로 전부 실행. 퇴근할 때 "중지" 누르면 역순으로 정리.
트레이에 상주하며 프로젝트별 레시피를 관리. 팀원에게 레시피 파일 공유 가능.

**왜 대체 불가**: 바로가기 모음? 순서 제어 불가. 배치 파일? 앱 열기/닫기 추적 불가.
Run.Deck은 "프로세스 라이프사이클 관리"까지 한다 — 내가 띄운 것만 골라서 정리.

### 2. Key.Tape — 매크로 녹화기

AutoHotkey는 강력하지만, 스크립트를 "코딩"해야 한다.
반복 작업을 자동화하고 싶은 순간은 매일 오는데, 매번 스크립트 짤 시간은 없다.

**Key.Tape는 VCR처럼 동작한다: Record → 행동 → Stop → Play.**

- ⏺️ 녹화 시작 (단축키)
- 키보드 입력, 마우스 클릭, 딜레이 전부 기록
- ⏹️ 녹화 종료
- 단축키 지정 → 이후 해당 키를 누르면 동일하게 재생
- 시각적 편집: 녹화된 스텝을 목록으로 보여주고, 드래그로 순서 변경/삭제/딜레이 조정

**실전 예시**:
- Excel에서 매일 같은 포맷으로 셀 정리 (5스텝 → 1키)
- 테스트 계정 로그인 (ID/PW 입력 → 로그인 버튼 클릭, 매일 수십 회)
- 파일명 패턴 변경 (우클릭 → 이름 바꾸기 → 접두사 추가 → Enter)

**왜 대체 불가**: AutoHotkey=코딩 필요. Power Automate=무겁고 복잡.
Key.Tape는 "녹화 버튼 하나"로 매크로를 만든다.

### 3. Snap.Diff — 즉석 비교기

"이 두 JSON 응답 뭐가 다르지?" — 브라우저에서 diff 사이트를 검색한다.
"이 두 설정 파일 차이가 뭐야?" — VS Code를 열어서 파일 두 개를 비교한다.
"이 스크린샷이랑 지금 화면이랑 뭐가 달라졌지?" — 눈으로 훑는다.

**Snap.Diff는 "두 개를 던지면 차이를 보여주는" 앱이다.**

- **텍스트 모드**: 좌/우 패널에 텍스트 붙여넣기 → 차이점 즉시 하이라이트
- **파일 모드**: 두 파일을 드래그 앤 드롭 → 사이드바이사이드 diff (구문 강조)
- **이미지 모드**: 두 이미지 드래그 → 픽셀 차이 오버레이 (빨간색으로 표시)
- **클립보드 모드**: "비교할 첫 번째 복사" → "비교할 두 번째 복사" → 자동 diff

단축키 한 번에 열리고, 비교하고, 닫는다. 그게 전부.

**왜 대체 불가**: VS Code diff=파일 필요. 웹 diff=브라우저 전환.
Snap.Diff는 클립보드 두 번이면 끝. 이미지 비교는 기존 도구에 거의 없음.

### 4. Repo.Radar — Git 저장소 레이더

개발 폴더에 프로젝트가 20개 있다. 어제 어떤 레포에서 커밋 안 하고 퇴근했는지 기억 안 난다.
브랜치를 만들어놓고 잊은 게 있을 수도 있고, push 안 한 커밋이 있을 수도 있다.
일일이 `cd` + `git status`? 20번?

**Repo.Radar는 모든 Git 저장소를 한 화면에 보여준다.**

```
📁 +Playground     main    ✅ Clean         ↑0 ↓0
📁 my-api          develop ⚠️ 3 uncommitted  ↑2 ↓0   ← push 안 한 커밋 2개!
📁 portfolio       main    ⚠️ 1 untracked    ↑0 ↓3   ← pull 필요!
📁 side-project    feature ✅ Clean         ↑0 ↓0
📁 dotfiles        main    ❌ Conflict       ↑1 ↓1   ← 충돌!
```

- 지정한 폴더(들)를 스캔해서 `.git`이 있는 레포 자동 발견
- 실시간 상태 갱신: uncommitted changes, unpushed commits, upstream 차이
- 더블클릭 → 해당 폴더에서 VS Code 또는 터미널 열기
- stale 브랜치 경고 (30일 이상 변경 없는 브랜치 표시)

**왜 대체 불가**: GitHub Desktop=레포별 관리. GitKraken=무겁고 유료.
"내 모든 레포를 한 눈에" 보여주는 가벼운 도구는 없다.

### 5. Proc.Pilot — 프로세스 사냥꾼

`Port 5000 is already in use.` — 개발자가 하루에 한 번은 보는 에러.
그러면 시작하는 여정: `netstat -ano | findstr 5000` → PID 찾기 → `taskkill /PID xxxx`
또는 작업 관리자를 열어서 수십 개 프로세스 중에서 찾기...

**Proc.Pilot은 "뭐가 이 포트 쓰고 있어?"에 1초 만에 답한다.**

- 🔍 검색창에 포트 번호 입력 → 점유 프로세스 즉시 표시 + 원클릭 종료
- 🔍 검색창에 프로세스 이름 입력 → 해당 프로세스의 포트/PID/메모리 표시
- 📊 전체 리스닝 포트 대시보드 (포트 번호순 정렬)
- ⚡ "자주 쓰는 포트" 모니터링 (3000, 5000, 8080 등) — 상태 변화 시 알림
- 🔒 포트 예약 기능: "5000번은 항상 내 API 서버용" → 다른 프로세스가 점유하면 경고

**왜 대체 불가**: netstat=CLI 기억력 테스트. 작업 관리자=포트 정보 없음. TCPView=UI가 2005년.
Proc.Pilot은 검색창 하나로 프로세스/포트 양방향 탐색.
